为什么选择golang（十）

7月 13 2021

前言：

我是万万没有想到今天碰见这道题了，其实我心里也算是有设想过很次了，但是今天突然问起来自己的思路还是很凌乱的，现在好好的从几个方面做个整理。

最后告诉自己一点自信很重要，我尼玛学了那么多东西，这两年干的拼死拼活的，我怕啥，我尼玛自己做了，应该理直气壮一点。不会的也要自信的告诉小爷就是不会，谁怎么可能都会。

1、第一点 golang给我一种安全感和不知名的优雅感。

我也在我这简短的两年的科研生活里，用过C++、Python、Java，C语言。golang就给我一种很厉害又很轻盈的感觉。 Go拥有强大的编译检查、严格的编码规范和完整的软件生命周期工具，具有很强的稳定性，稳定压倒一切。Go提供了软件生命周期（开发、测试、部署、维护等等）的各个环节的工具，如go tool、gofmt、go test。

2、golang的执行性能

golang在语言层面支持了高并发—Goroutine，并发一般就是CPU密集型和IO密集型，golang就是IO密集型。通过CPU把IO命令交给IO接收器，然后CPU继续处理IO指令，不用等待IO指令完成，IO接收器设备异步处理。这样实现了大的并发。其次就是语言级别的支持的Goroutine协程，还是在用户态支持并发，Goroutine 上下文切换只涉及到三个寄存器（PC / SP / DX）的值修改；而对比线程的上下文切换则需要涉及模式切换（从用户态切换到内核态）、以及 16 个寄存器、PC、SP…等寄存器的刷新。线程栈空间通常是 2M，Goroutine 栈空间最小 2K；这样的优点就让它很适合在服务器端进行开发。

3、开发效率高

语言简单，我觉得它就是互联网界的C语言，有很多的开源的框架，有很多写好的包，和标准库。而且支持编译成2进制，然后打包也方便，或者用docker打包嘛，docker就是go开发的。我觉得它特别适合拿来搭建微服务，golang从开发到测试到部署整个都很快速和简单。

4、前景

我学这个的驱动力是区块链，后来加入了云平台，让我开始想学习这个语言。这个语言有谷歌作为背书，所适用的场景也是未来最有前途之中的两个场景，我觉得golang不是昙花一现，我觉得未来很精彩。

  • 10、为什么是go

    • 1、强大的标准库

      • 这包括互联网应用、系统编程和网络编程。Go里面的标准库基本上已经是非常稳定了，特别是我这里提到的三个，网络层、系统层的库非常实用。

    • 2、部署、运行

      • Go 的程序，编译出来即为一个2进制文件，可以直接运行，不需要安装其他依赖，非常方便
        运行时，占用的初始内存极低，同样的功能，如果程序写得好的话，相比其他语言占用的内存都少，也就意味着，需要消耗的计算资源也越少

    • 3、因为 Go 很“快”。简单的并发

      • 1、背景

        • 随着信息技术的迅速发展，单台服务器处理能力越来越强，迫使编程模式由从前的串行模式升级到并发模型。
        • 并发模型包含 IO 多路复用、多进程以及多线程，这几种模型都各有优劣，现代复杂的高并发架构大多是几种模型协同使用，不同场景应用不同模型，扬长避短，发挥服务器的最大性能。
        • 而多线程，因为其轻量和易用，成为并发编程中使用频率最高的并发模型，包括后衍生的协程等其他子产品，也都基于它。

      • 2、Golang 为并发而生

        • 1、Golang 从 2009 年正式发布以来，依靠其极高运行速度和高效的开发效率，迅速占据市场份额。Golang 从语言级别支持并发，通过轻量级协程 Goroutine 来实现程序并发运行。

          • 语言界别的支持

            • 说一种语言提供语法来支持这样的行为
            • 比如nodejs天生就是异步的，这样就和编写同步代码差不多了

        • 2、Goroutine 非常轻量体现在两个方面

          • 1、上下文切换代价小：

            • 1、Goroutine 上下文切换只涉及到三个寄存器（PC / SP / DX）的值修改；而对比线程的上下文切换则需要涉及模式切换（从用户态切换到内核态）、以及 16 个寄存器、PC、SP…等寄存器的刷新；

          • 2、内存占用少：

            • 线程栈空间通常是 2M，Goroutine 栈空间最小 2K；

          • Golang 程序中可以轻松支持10w 级别的 Goroutine 运行，而线程数量达到 1k 时，内存占用就已经达到 2G。

            • 它提供了海量并行的支持，这对于游戏服务端的开发而言是再好不过了。

          • 3、Go 调度器实现机制：

            • Go 程序通过调度器来调度Goroutine 在内核线程上执行，但是 G - Goroutine 并不直接绑定 OS 线程 M - Machine运行，而是由 Goroutine Scheduler 中的 P - Processor （逻辑处理器）来作获取内核线程资源的『中介』。

            • G-P-M 模型

              • 通过该模型怎样实现少量内核线程支撑大量 Goroutine 的并发运行。

            • 以及通过 NetPoller、sysmon 等帮助 Go 程序减少线程阻塞，充分利用已有的计算资源，从而最大限度提高 Go 程序的运行效率。

          • 4、goroutine 创建简单，系统消耗小，channel 的机制方便 goroutine 之间的通信，可以很轻松的来实现一个高并发的服务

    • 3、稳定性

      • Go拥有强大的编译检查、严格的编码规范和完整的软件生命周期工具，具有很强的稳定性，稳定压倒一切。那么为什么Go相比于其他程序会更稳定呢？这是因为Go提供了软件生命周期（开发、测试、部署、维护等等）的各个环节的工具，如go tool、gofmt、go test。

    • 4、适合做什么

      • 1、Go语言主要用作服务器端开发，其定位是用来开发“大型软件”的，适合于很多程序员一起开发大型软件，并且开发周期长，支持云计算的网络服务。

      • 2、云计算、云生态大部分都是采用go实现

        • Go 比较适合的场景是做一些服务中间件，比如消息队列，服务网关等需要支持到高并发的场景
        • docker、k8s都是用go写的

      • Go语言作为服务器编程语言，很适合处理日志、数据打包、虚拟机处理、文件系统、分布式系统、数据库代理等；

      • 网络编程方面，Go语言广泛应用于Web应用、API应用、下载应用等；

    • 5、语言特性

      • 推导类型，即变量不一定需要来定义类型，类型由编译器来推导确定，这是在很多新语言里都涌现的功能(比如 Kotlin 和 Swift)，避免很多的类型定义，能大大提高编程效率
      • 支持类型反射，可以针对一些数据结构做通用话的处理，可以方便的做序列化，反序列化（比如自带的 json Marshal/Unmarshal 功能），在自己的业务里，可以实现类型的功能，来对数据结构做一层序列化、反序列化的封装
      • field tag（不知道怎么翻译成中文好），可以实现对struct字段的自动处理，比如以上提到的 json Marshal/Unmarshal 的功能，可以定义最后序列化成的字段名是怎样的，也可以定义字段为空时，是否要包含该字段；类似的，我们可以自己定义 tag 类型，定义 tag 的语义，来满足自己的需求
      • 支持多个返回参数，在我之前写 Python 的时候，很喜欢的一个语言特性，现在在 Golang 里用到也很得心应手
      • 内置 GC，大部分时候都不需要手动去处理内存的分配与释放

    • 6、还有很多web框架iris，gin等

    • ——-最终回答——-

      • 优点

        • 拥有轻量的应用层进程 goroutine，允许开发者基于大量 goroutine 来设计并发程序
        • First class channel 的支持，使得 goroutine 之间能够很轻易地相互合作
        • select 关键字的随机能力使得开发者可以基于 channel 来对程序实现调度
        • 使用返回值的形式处理错误，很好地契合了 goroutine + channel 的并发模型
        • 库程序里有我想要的一切，总体实现较为完善。学习体验也十分顺畅，不得不说，Go 是一种经过精心设计的实用性语言。

      • 缺点

        • 泛型的缺失导致许多程序设计变得脆弱，增加代码量且失去了安全性。

        • 代码中大量充斥着 interface{} 和对 interface{} 的类型转换。其原因就是我们在实现这样一个服务模板时，我们并不知道模板的使用者需要处理怎样的 request，返回怎样的 response，也不知道这里的 state 是什么。

        • 由于泛型的缺失，我们的代码相当于失去了静态的类型检查，将静态的类型错误变为了运行时错误，这样一来，Go 语言的静态能力就缺失了很多，甚至我们可以说，Go 语言泛型的缺失使得 Go 语言在类型安全性上不如带有泛型能力的静态类型语言，却比这些语言在使用上还要更啰嗦（各种类型转换和错误判断）。
